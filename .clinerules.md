# BBplay Project Context for AI Assistants (clinerules)

This document provides all necessary context for generating code for the "BBplay" Flutter application.

## 1. Project Goal
To create a cross-platform mobile application for the "BBplay" computer club network. The app allows users to register, log in, top up their balance, book gaming PCs, and interact with a support chat bot.

## 2. Core Architecture & Technologies
* **Framework:** Flutter (Dart 3+)
* **State Management:** `Provider` package (`ChangeNotifier`). All global states (user data, settings) are managed through providers registered in `main.dart`.
* **Navigation:** `CurvedNavigationBar` for the main layout. Navigation logic is managed by `MainLayout` using a `PageController`.
* **Networking:** `http` package with a custom `ApiClient` class (`lib/core/utils/api_client.dart`).
* **Storage:** `shared_preferences` for persisting user settings (theme, language).
* **UI Libraries:** `font_awesome_flutter` (icons), `google_fonts` (typography, main font is Montserrat).
* **Code Style:** Strict linting rules defined in `analysis_options.yaml` (based on `flutter_lints`).

## 3. API & Server Logic
* **Base URL:** `https://vibe.blackbearsplay.ru`
* **Data Format:** All `POST` requests must send data in `application/json` format.
* **Authentication:**
  * Login endpoint: `/login` (POST). Expects `member_name` and `password`.
  * Registration: 3-step process (`/api/v2/.../members`, `/request-sms`, `/verify`).
* **Key Endpoints:**
  * `/cafes` - Get club list.
  * `/struct-rooms-icafe` - Get room structure and PC layout.
  * `/available-pcs-for-booking` - Check PC availability for a specific time.
  * `/booking` - Book a PC (requires `rand_key` and `key` MD5 signature).
  * `/all-books-cafes` - Get user's booking history.
* **Server Quirks (IMPORTANT):**
  * The server is unstable and its documentation is often incorrect.
  * Success codes can be `0`, `3`, or `200`. The word "Success" can be misspelled as `Succes`.
  * The server may return an "error" message (e.g., "Incorrect parameter") with a "success" code (`code: 0`). **Always check the `message` field for error keywords.**
  * The server does not correctly report PC coordinates. We use a static grid layout (`Row` + `Expanded`), ignoring `pc_box_left/top`.
  * The `/topup` (balance) and `/cancel` (booking) endpoints may be blocked (`Api not allowed`). The app must handle this gracefully with a fallback UI.

## 4. File Structure
* `lib/main.dart`: Entry point, `MaterialApp` setup, `MultiProvider` registration.
* `lib/providers/`: Contains all `ChangeNotifier` classes (`user_provider.dart`, `settings_provider.dart`).
* `lib/screens/`: Contains all full-page widgets (`booking_screen.dart`, `profile_screen.dart`, etc.).
* `lib/core/utils/`: Core utilities, including `api_client.dart`.
* `lib/widgets/`: Reusable UI components (e.g., `video_player_widget.dart`).

## 5. Coding Rules & Patterns
* **Stateless over Stateful:** Prefer `StatelessWidget` where possible. Use `Provider` for state management instead of local `StatefulWidget` state for anything that needs to be shared.
* **Localization:** All user-facing strings must be retrieved via `Provider.of<SettingsProvider>(context).getText('key')`. Do not use hardcoded strings.
* **Theming:** All colors, fonts, and styles must be derived from `Theme.of(context)`. Do not use hardcoded colors like `Colors.white` or `Color(0xFF...)`.
    * Background: `Theme.of(context).scaffoldBackgroundColor`
    * Card/Container color: `Theme.of(context).colorScheme.surface`
    * Text color: `Theme.of(context).colorScheme.onSurface`
    * Accent color: `Theme.of(context).primaryColor`
* **Error Handling:** All API calls must be wrapped in `try-catch` blocks. The UI must never crash due to a server error. Always show a `SnackBar` or a user-friendly error message.
* **Performance:** Use `const` constructors for widgets wherever possible to improve performance.
* **Readability:** All public methods and classes should have a brief documentation comment.
